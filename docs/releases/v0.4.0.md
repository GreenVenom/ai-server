# v0.4.0 - Qdrant Vector Database

## Summary

M04 adds a production-quality local vector database to the Personal AI Platform.

Qdrant 1.18.2 now runs as a version-pinned Docker Compose service with loopback-only networking, durable named-volume storage, automatic recovery after Docker Desktop and machine lifecycle events, tested Ollama embeddings, deterministic metadata conventions, semantic retrieval, validated snapshot recovery, retention tooling, and platform-wide health verification.

## Highlights

- Qdrant 1.18.2 deployed through Docker Compose.
- REST bound to `127.0.0.1:6333`.
- gRPC bound to `127.0.0.1:6334`.
- Live data stored in `personal-ai-qdrant-storage`.
- `restart: always` validated after Docker Desktop restart and reboot.
- `nomic-embed-text:latest` validated at 768 dimensions.
- Named vector `text-dense` with Cosine distance.
- Five-point semantic validation corpus retained.
- Deterministic UUIDv5 identity and payload schema standardized.
- Native snapshots downloaded with SHA-256 manifests.
- Clean restore reproduces schema, payloads, vectors, and ranking.
- Internal and portable snapshot retention automated.
- Qdrant integrated into `status.sh`, `health.sh`, and `verify.sh`.

- ADR-0013 — Standardize Qdrant Snapshot Retention and Cleanup

## Breaking or Behavioral Changes

- Qdrant uses `restart: always`, not `unless-stopped`.
- Docker Desktop restart will automatically restore Qdrant.
- An intentionally stopped Qdrant container may start when Docker Desktop next starts.
- Live data resides in a Docker named volume rather than a host bind directory.

## Known Constraints

- Docker services become available only after the `openclaw` user logs in.
- Qdrant is local-only and currently has no API key.
- Portable backups are local to the same Mac until M08 expands disaster recovery.
- Obsidian ingestion and OpenClaw retrieval integration are deferred to later milestones.
